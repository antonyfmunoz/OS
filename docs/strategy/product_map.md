## UMH — Universal Meta Harness

| Field | Value |
|-------|-------|
| **Purpose** | Universal intelligence substrate — powers all OS products |
| **Type** | Substrate / framework / kernel — NOT a SaaS UI |
| **User/Customer** | Internal (OST products) — never sold directly |
| **Internal vs Public** | Internal only — consumed by EOS, CreatorOS, LYFEOS |
| **First Use Case** | Power EntrepreneurOS to run first operating workflow |
| **Long-Term Role** | Foundation for AI OS, proprietary runtime, robotics-control |
| **Data Produced** | Architecture phases, ontology laws, governance policies, execution traces, adapter configs, registry items, storage records |
| **Connection to EOS** | UMH IS the substrate under EOS — EOS is the product layer on top of UMH |

**Includes**: governed execution spine, ontology/law kernel, memory/second-brain layer, registry/templates/libraries, adapter isolation, typed contracts, storage discipline, observability, deliberation council, interface/command center.

**Modular**: AI, harness, and OS can be separated. A user could use the harness without AI, or the AI without the OS shell.

| Manufacturing Intelligence | Production planning, quality tracking, yield optimization, machine utilization, supply chain management, manufacturing memory | After first physical production |
| Robotics / Actuator Component Intelligence | Component selection, actuator specification, custom part design briefs, assembly optimization, firmware management, maintenance prediction | After robotics capability established |
| Supplier / Prototype Evaluation | Supplier discovery, quote comparison, lead time tracking, quality scoring, prototype iteration management, DFM feedback loops | After first physical product development |

**Governing doctrines for physical products** (Strategic Context Amendment v2):
- Anti-Copycat Moat Doctrine — product features alone are not defensible; moat requires proprietary data, distribution, physical infrastructure, manufacturing, robotics, and operating history
- Physical Infrastructure Moat Doctrine — progression from software to physical production to manufacturing to robotics to capital allocation
- Custom Actuator / Robotics Component Doctrine — buy commodity, customize strategic, own bottleneck, manufacture moat-critical actuator and component parts; CNC machining and prototyping only where ownership creates advantage
