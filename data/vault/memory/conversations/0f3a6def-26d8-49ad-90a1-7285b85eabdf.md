`★ Insight ─────────────────────────────────────`
**Memory vs Storage distinction** — This is one of the most common conflation errors in system design. Memory is an *intelligence* subsystem: it has semantic recall ("what do I know about X?"), episodic retrieval ("what happened last time I did Y?"), and working memory ("what's relevant right now?"). Storage is *infrastructure*: it has `get(key)`, `put(key, value)`, `all_keys()`. A memory system *uses* storage, but collapsing them into one subsystem means you lose the ability to reason about recall policies, forgetting, and relevance ranking independently from persistence mechanics.

**Adapter discovery centralization** — The four interface files (`goals/interfaces.py`, `strategy/interfaces.py`, `memory/storage.py`, `execution/interfaces.py`) all repeat the exact same `try: from eos_ai.adapters...; except ImportError: return Null*` pattern. This is the classic "optional dependency binding" pattern from plugin architectures. Centralizing it into a single `bridge.py` means platform-specific import paths change in one place, not four.

**Why decontaminate before renaming** — If you rename packages while contaminated imports still exist, the import rewrites touch the same files that need decontamination surgery. You're editing a moving target. Fix the contamination first (surgical, high-risk), then do the renames (mechanical, low-risk). Sequential risk reduction, not parallel.
